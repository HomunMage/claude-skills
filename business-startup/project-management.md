## Core Concept: Pyramid Hierarchy

Like 憲法→法律→條則 (Constitution → Law → Rules), every project needs a strict hierarchy:

```
目的 (Purpose)        ← Why does this project exist?
  └─ 願景 (Vision)    ← What does the finished product look like?
      └─ 設計支柱 (Design Pillars) ← How do we achieve the vision? (rank by priority)
          └─ 系統 (Systems)        ← What specific systems/features fulfill each pillar?
              └─ 細節 (Details)    ← Implementation details
```

- **Every decision traces back up the hierarchy**
- When conflicts arise, higher-level priorities win
- The team MUST align on this hierarchy before any development starts
- Regular reviews to check if the hierarchy still holds

## Early Validation (最重要)

**Never build without validation first:**

1. **Concept Validation (POC)** — prove the idea works with minimum effort (paper prototype, one chapter, one boss fight)
2. **Technical Validation** — confirm the team can actually build it
3. **Market Validation** — get people to PAY before full development (fundraise first, pre-orders, paid subscriptions)
4. **Continuous POC/MVP** — keep validating at every stage, not just the beginning

## Development Process: Agile + Spiral

### Milestone Structure
```
Brainstorming → PoC → Whitebox → Gameplay → Vertical Slice → Mass Production → Polish
```

### Key Principles

- **Vertical Slice is critical** — build one section to near-ship quality EARLY
  - First time team encounters ALL aspects of finishing the product
  - If it's not fun at Vertical Slice, there's a fundamental problem
  - Good for finding investors/publishers, building demos, marketing materials
  - Only projects that pass Vertical Slice are truly in production

- **Concentric development** — build from design pillars outward
  - Core systems first (minimum to achieve each pillar)
  - Only expand outward after core is solid
  - When cutting scope, cut from outermost ring inward
  - Have multiple plans for different scope levels (MVP mindset)

- **Time estimates**: add 20%+ buffer for unknowns; unfamiliar work needs even more
- **Wishlist**: things that don't fit go to a wishlist for when time frees up
- **Polish phase**: reserve 20-25% of total dev time; no new features, only bug fixes and polish

### Pre-Production Documents
1. **Game Design Document** (GDD) — vision, pillars, systems, gameplay, story, art style
2. **Milestone & Work Plan** — break pillars → systems → tasks, schedule with buffers
3. **Budget & Cash Flow** — track ALL expenses with reasons; everyone gets paid (even minimum wage)

## Accelerating Development

### Meetings
- **Before**: Clear topic, estimated duration, advance notice, only relevant people required
- **During**: State impact of decisions, one topic at a time, clarify purpose before discussing methods, use visuals/examples
- **After**: Record decisions AND reasoning, centralize meeting notes

### Decision-Making
- **Reversible vs Irreversible** — only irreversible decisions need deep thought
- Reversible decisions: decide fast, try immediately, learn from results
- Voting: list all options → discuss pros/cons → unlimited votes → start from top vote → if tied, producer decides

### Team Ownership
- Give team members Sense of Ownership (not just executing someone else's vision)
- Open discussion + fair decision process → trust
- Welcome challenges, face problems honestly
- Team-level accountability for results (don't blame the idea proposer)
